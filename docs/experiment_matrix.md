accounting at small scale. They do not prove agency, OOD generalization, or
broad exact-value learning.

## Notes

- Exact rows and rejected rows are separated by `label_kind`; rejected rows are never used as `exact.value_class` supervision.
- The value-class baseline is now multi-class but intentionally tiny: it exposes
  the minority `switch` class and records that a majority baseline misses it.
- Wave 6 scales deterministic row generation and rejection accounting; its 0.2
  FEN-gate accuracy is a useful negative control, not a model result.
- Wave 6 and Wave 7 position-hash reports are now explicit leakage audits:
  they catch D4-symmetric positions crossing splits.
- The symmetry-hash reports are the leakage-safe split artifacts for model
  scoring; position-hash reports should not be used for headline evaluation.
- Wave 9 adds the current evaluation-ready KRK-family holdout protocol:
  KRK is test-only, KQK train/dev is D4-symmetry-safe, and no model has been
  trained on it yet.
- Wave 10 scores the fixed FEN-gate floor on the Wave 9 split; any learned
  model result must report against that same split and baseline.
- Wave 11 adds a hand-feature logistic probe on the same split. It is a useful
  baseline to beat, not a substitute for structured-label or agency evidence.
- Wave 12-14 expand material-family breadth to KQK/KRK/KBK/KNK and show that a
  hand geometry probe can solve the KNK holdout, making non-number targets and
  harder compositional splits the next pressure point.
- Wave 15 adds the first exact-only frontier metadata target report. It exposes
  a KRK-family OOD shift in `frontier_mean`, but frontier temperature remains
  constant on this shard.
- Wave 17 is the executable composition-certificate dispatch contract. It is
  not yet a dataset or model result; it defines the gates required before
  decomposition-benefit claims can be tested.
- Current metrics are plumbing checks for the current artifacts and should not
  be used as evidence beyond those artifacts.
- Negative-control rows are not training data. They define expected failures for schema, domain, decomposition, and exact-training gates.
